# 버튜버 스타일 실시간 방송 방 서비스 — 프로젝트 스펙

> 방장의 웹캠 표정이 아바타로 실시간 반영되고, 참가자들이 채팅·음성으로 소통하는 다중 방(room) 서비스

---

## 1. 프로젝트 개요

버튜버 스타일 실시간 방송 방 서비스. 방장 1명과 시청자 최대 4~5명이 한 방에 모여 소통한다.

- 방장의 웹캠 표정이 아바타(VRM)로 실시간 반영된다.
- 참가자들은 텍스트 채팅과 음성으로 소통한다.
- 방은 여러 개 생성 가능하며, "방 목록 → 입장" 구조를 따른다.
- 한 방의 정원은 방장 포함 5~6명 수준.

---

## 2. 핵심 설계 결정

이 세 가지 판단이 전체 기술 스택을 결정한다.

### 2-1. 방장의 실제 웹캠 영상은 전송하지 않는다
브라우저에서 얼굴을 트래킹해 **표정 계수(52개 블렌드셰이프)와 머리 회전값**만 뽑아 전송하고, 아바타는 각 참가자의 브라우저가 직접 렌더링한다. 웹캠 원본 영상은 서버로 나가지 않는다.
- 장점: 대역폭이 극히 가볍다(프레임당 1KB 미만), 아바타 화질이 시청자 기기 성능에 따라 무한대, 프라이버시 우수.

### 2-2. 표정·채팅은 WebSocket, 음성만 WebRTC
- **표정 데이터 / 채팅**: 초당 1KB 미만 → 일반 WebSocket(STOMP) 브로드캐스트로 충분.
- **음성**: 실시간성이 중요 → WebRTC P2P로 처리.

### 2-3. SFU(미디어 서버)는 필요 없다
인원이 5~6명 고정이고 무거운 영상 스트림이 없으므로 **음성 Mesh만으로 감당**된다.
- 음성 트랙은 30~50kbps 수준 → 방장이 시청자 4명에게 4개 업로드해도 약 200kbps.
- 별도 미디어 서버 없이 기존 Spring Boot 스택 안에서 대부분 해결.

---

## 3. 데이터 흐름

| 데이터 종류 | 전송 방식 | 경로 | 특징 |
|---|---|---|---|
| 표정 데이터 | WebSocket (STOMP) | 방장 → 서버 → 시청자 | 초당 30fps, 프레임당 1KB 미만 |
| 채팅 / 후원 | WebSocket (STOMP) | 양방향, 서버 경유 | MySQL에 저장 |
| 음성 | WebRTC | 방장 ↔ 시청자 P2P (Mesh) | 트랙당 30~50kbps |

- 표정 브로드캐스트 토픽 예시: `/topic/room/{roomId}/face`
- 채팅 브로드캐스트 토픽 예시: `/topic/room/{roomId}/chat`

## 4. 확정 기술 스택

### 프론트엔드
- **React + Vite + TypeScript**
- 상태 관리: **Zustand** (참가자 목록, 음소거 상태, 발화자 등 UI 상태) + **TanStack Query** (방 목록/데이터 페칭)
- WebSocket 클라이언트로 표정·채팅 처리
- WebRTC(`RTCPeerConnection`)로 음성

### 얼굴 트래킹
- **MediaPipe Face Landmarker** (Google, 브라우저에서 WASM+GPU로 동작, 무료)
- 웹캠 → 478개 랜드마크 + 52개 블렌드셰이프 + 머리 회전값 추출

### 아바타 렌더링
- **VRM (3D) 포맷** — 무료 모델(VRoid Hub) 풍부, 라이선스 깔끔
- `@pixiv/three-vrm` + **Three.js** 로 렌더링
- **kalidokit** 으로 MediaPipe 랜드마크 → VRM 본/블렌드셰이프 매핑
- (대안: Live2D + `pixi-live2d-display` — 2D 스타일이지만 모델 확보·Cubism 라이선스 확인 필요)

### 백엔드
- **Spring Boot**
- **STOMP (WebSocket)**: 표정·채팅 브로드캐스트, WebRTC 음성 시그널링(같은 채널에 메시지 타입으로 얹음)
- **REST**: 방 관리(CRUD), 인증(JWT)

### 데이터베이스
- **MySQL / JPA**

### 미디어 보조
- **coturn** (TURN 서버) — NAT 통과용, 음성 단계에서 Docker로 추가

### 인프라
- **Docker** → **AWS EC2** + **RDS(MySQL)**
- VRM 파일은 **S3** 에 저장, 입장 시 다운로드 후 캐싱 (모델당 10~30MB)
- **HTTPS/WSS 필수** (웹캠·마이크 접근 조건)

## 5. 구현 순서

### 1단계 — 방 시스템 + 채팅
- 방 CRUD REST API
- STOMP 채팅
- 방 목록 / 입장 페이지
- 순수 Spring Boot + React, 미디어 없음

### 2단계 — 아바타 로컬 렌더링 *(핵심 검증 단계)*
- 방장 화면에서 웹캠 → MediaPipe → kalidokit → three-vrm
- 자기 아바타가 표정 따라 움직이는 것까지 (네트워크 없이 브라우저 안에서만)
- 이 프로젝트에서 가장 낯설고, 동작하는 순간 전체가 실감 나는 단계

### 3단계 — 표정 데이터 브로드캐스트
- 2단계 데이터를 WebSocket으로 전송
- 시청자가 같은 VRM을 받은 데이터로 구동
- 채팅 인프라 재사용

### 4단계 — WebRTC 음성
- 방장 ↔ 시청자 음성 Mesh
- 시그널링은 기존 WebSocket에 메시지 타입 추가
- coturn 붙임

### 5단계 — 폴리싱
- 후원 알림 오버레이 (금액 표시 등)
- 방장 화면 공유 (`getDisplayMedia`, 640p 정도)
- 볼륨 기반 립싱크 보정 (음성 볼륨 → `aa` 블렌드셰이프 가중치)

## 6. DB 스키마 (초안)

| 테이블 | 주요 컬럼 |
|---|---|
| `users` | id, nickname, ... (기존 인증 스키마) |
| `rooms` | id, title, host_user_id, max_participants, status |
| `room_participants` | room_id, user_id, role(HOST/VIEWER), joined_at |
| `chat_messages` | room_id, user_id, content, type(CHAT/DONATION), created_at |
| `avatars` | user_id, vrm_url |

- "방 여러 개"는 인프라 문제가 아니라 **네임스페이스 문제**다. STOMP 토픽을 `/topic/room/{roomId}/...` 로 나누고, 서버 메모리(또는 Redis)에 방별 참가자 세션 목록을 관리하면 된다.

## 7. 알아둘 함정

- **HTTPS/WSS 필수**: `getUserMedia`는 보안 컨텍스트에서만 동작. `localhost`는 예외라 개발은 되지만, EC2 배포 시 반드시 SSL(Let's Encrypt + Nginx 등) 필요. ("로컬은 되는데 배포하면 카메라 안 켜짐" 함정)
- **AWS 보안 그룹의 UDP 포트**: WebRTC 음성·TURN은 UDP 사용. coturn/음성이 쓰는 UDP 포트 범위를 열어야 함.
- **표정-음성 싱크**: 표정(WebSocket)과 음성(WebRTC)이 다른 경로로 오므로 미세한 어긋남 가능. 표정 프레임에 **타임스탬프**를 실어 시청자 쪽 **100~200ms 버퍼**로 맞춤.
- **저사양 기기 부담**: 시청자 기기에서 3D 렌더링이 돌아가므로 아주 저사양 기기에선 부담될 수 있음.
